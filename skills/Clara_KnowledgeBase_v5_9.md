# Clara — Knowledge Base v5.9

Referencia para Clara. Tablas estáticas, rúbricas de madurez 1-9, banco de preguntas de verificación, plantilla del DBI y mapeo al Scorecard de Innovación. Clara captura; el Evaluador puntúa; el comité decide.

## Rúbricas de madurez 1-9 (Clara infiere; el usuario nunca se autoclasifica)
Regla acumulativa: el nivel es el más alto cuyos requisitos —y los de todos los inferiores— están demostrados con evidencia. Ante duda persistente, el nivel menor. Fuente: Calculadora de Nivel de Maduración (adaptada) + Anexo TRL ANID.

### TRL — madurez técnica (misma lectura Interna/Externa)
| Nivel | La evidencia demuestra que... |
|---|---|
| 1 | Se observaron o estudiaron los principios básicos; solo existe la idea. |
| 2 | Hay una aplicación específica formulada para un problema concreto; quizás modelos o análisis preliminares, sin pruebas. |
| 3 | Hubo prueba de concepto experimental o analítica de los componentes o parámetros críticos. |
| 4 | Los componentes clave fueron validados funcionando integrados en laboratorio o entorno controlado. |
| 5 | Existe un prototipo inicial probado en condiciones que simulan representativamente el entorno real. |
| 6 | El prototipo fue demostrado en un entorno realista (condiciones cercanas a las reales, usuarios o procesos reales, escala pre-comercial). |
| 7 | El sistema completo (o prototipo muy avanzado) fue validado en operación real con desempeño confiable. |
| 8 | La tecnología está finalizada y su desempeño fue calificado formalmente (certificación, verificación, estándar del sector). |
| 9 | El sistema está integrado en su entorno operativo definitivo, en uso continuo con resultados repetibles. |

### CRL — validación del cliente (lectura según Tipo)
| Nivel | Externa (cliente/mercado) | Interna (área usuaria) |
|---|---|---|
| 1 | Hipótesis especulativa de una necesidad; sin conversaciones. | Cree que un área podría tener el problema; no lo ha conversado. |
| 2 | Revisó fuentes secundarias; problema descrito con más especificidad. | Revisó datos o reportes internos que describen el problema. |
| 3 | Primer feedback: contacto con usuarios o expertos valida el problema. | Conversó con el área afectada y esta confirma el problema. |
| 4 | Problema confirmado con múltiples usuarios; segmentación inicial. | Varias áreas o plantas confirman; sabe exactamente a quién aplica. |
| 5 | Interés concreto de actores clave; aportan requerimientos funcionales. | El área pidió avanzar o aportó requerimientos concretos. |
| 6 | Beneficios confirmados en pruebas o colaboraciones iniciales con usuarios. | Prueba conjunta con el área con evidencia de beneficio. |
| 7 | Pruebas extendidas o pilotos formales con clientes; validación en entorno real. | Piloto formal en curso con el área usuaria, con resultados. |
| 8 | Ventas iniciales o disposición de pago validada; adopción estructurándose. | El área adoptó la solución en parte de su operación y comprometió recursos. |
| 9 | Adopción consolidada y repetible en múltiples usuarios reales. | Uso repetible en operación; demanda interna consolidada. |

### BRL — madurez del modelo (lectura según Tipo)
| Nivel | Externa (modelo de negocio) | Interna (caso de ahorro/valor) |
|---|---|---|
| 1 | Hipótesis muy preliminar; no es claro cómo genera valor. | No es claro cómo genera ahorro o valor. |
| 2 | Problema y propuesta de solución descritos con claridad. | Lógica de ahorro descrita, sin números. |
| 3 | Borrador de modelo: mercado potencial y competencia descritos. | Lógica de ahorro con alternativas consideradas y primeras cifras. |
| 4 | Primera versión completa con ingresos y costos estimados. | Caso completo: costo de implementar y ahorro anual estimados, con fuente. |
| 5 | Elementos del modelo evaluados con usuarios reales; modelo actualizado. | Estimaciones contrastadas con el área o con datos reales de operación. |
| 6 | Precios o costos confirmados por usuarios reales vía piloto o venta técnica. | Ahorro confirmado con datos de una prueba real. |
| 7 | Ajuste solución-mercado comprobado; disposición real de pago. | El área confirma que resuelve mejor que las alternativas y compromete presupuesto. |
| 8 | Ventas o adopciones con métricas que confirman sostenibilidad y escala. | Adopción con métricas que confirman el ahorro de forma sostenida. |
| 9 | Modelo definitivo, estable y en expansión. | Caso de valor consolidado y replicándose en otras áreas/plantas. |

## Banco de preguntas de verificación (una por nivel; usar SOLO ante ambigüedad entre niveles adyacentes, adaptada a lenguaje natural y al contexto)
| Nivel | TRL | CRL (adaptar Int/Ext) | BRL (adaptar Int/Ext) |
|---|---|---|---|
| 2 | ¿Tienes definida una aplicación concreta, con algún modelo o análisis que anticipe cómo funcionaría? | ¿Has revisado datos, estudios o reportes que describan el problema? | ¿Puedes describir el problema y cómo tu solución generaría valor? |
| 3 | ¿Hiciste alguna prueba de concepto, aunque fuera analítica o a escala de laboratorio? | ¿Conversaste con quien tiene el problema y te lo confirmó? | ¿Conoces las alternativas que existen y cómo se compara tu propuesta? |
| 4 | ¿Probaste los componentes funcionando juntos en un entorno controlado? | ¿Más de un usuario o área te confirmó el problema? | ¿Tienes el caso completo con costos e ingresos (o ahorro) estimados? |
| 5 | ¿Existe un prototipo probado en condiciones que simulan el entorno real? | ¿Alguien con poder de decisión mostró interés concreto o pidió requerimientos? | ¿Contrastaste tus números con usuarios o datos reales? |
| 6 | ¿Lo probaste en condiciones cercanas a las reales, con usuarios o procesos reales? | ¿Hubo una prueba conjunta donde se vio el beneficio? | ¿Los costos o precios fueron confirmados en una prueba real? |
| 7 | ¿El sistema completo operó en condiciones reales con buen desempeño? | ¿Hay un piloto formal en curso o pruebas extendidas con el cliente/área? | ¿Hay disposición real de pago o compromiso de presupuesto? |
| 8 | ¿Está certificado o calificado formalmente para operar? | ¿Hubo compra inicial o adopción con recursos comprometidos? | ¿Las métricas de la adopción confirman que el caso se sostiene? |
| 9 | ¿Está en uso continuo en operación, con resultados repetibles? | ¿La adopción es repetible y con varios usuarios reales? | ¿El modelo está estable y expandiéndose? |

## Otras escalas de referencia (solo para el DBI final)
**Rangos de impacto (USD/año)**: Interna Bajo <50k / Medio 50k-200k / Alto >200k. Externa Bajo <200k / Medio 200k-1M / Alto >1M. Clasificar todo monto, exacto o estimado.
**Escalabilidad**: Local (mi área) / Interna (otras plantas) / Externa (clientes o mercados nuevos). Potencial, no realidad; ante duda, lo más conservador consistente.

## Tamaño de mercado — estructura y mapeo
Los cuatro factores de B.2b se registran SEPARADOS, cada uno con fuente o supuesto:
- Mercado total (nº empresas + tamaño anual) ≈ TAM.
- Alcanzable Elecmetal (nº + razón) ≈ SAM.
- Clientes cerrables ~3 años (nº) y ticket anual → SOM = clientes × ticket.
Clara NO calcula el agregado; entrega factores. "Clientes cerrables" es el más especulativo: registrar como supuesto a validar salvo respaldo comercial.

## Campos derivados (bloque D) — asignados por innovación, NO se preguntan
- Foco estratégico: economía circular/sostenibilidad; soluciones digitales para minería; nuevos negocios y mercados.
- Horizonte: H1 incremental (mejora del core, capacidades actuales, baja incertidumbre); H2 adyacente (nuevo segmento/cliente/aplicación cercano al core); H3 transformacional (nuevo negocio/modelo, alta incertidumbre). Más incertidumbre tiende a H2/H3.

## Mapeo DBI → Scorecard (22 ítems puntuables, escala 1/3/5)
Para que el Evaluador puntúe con evidencia del DBI. Cada ítem cita el campo que lo alimenta.

| Dimensión (peso) | Ítem scorecard | Campo del DBI |
|---|---|---|
| Problema (15) | Claridad del problema | A: problema |
| | Relevancia/dolor | A: por qué importa |
| | Interés de quien lo tiene | A: quién lo tiene + C: CRL |
| | Competencia/sustitutos | B: competencia |
| Solución (20) | Claridad de la solución | B: descripción |
| | Diferenciación/novedad | B: diferenciador (grado) |
| | Estado de desarrollo/TRL | B.3: TRL |
| | Impacto esperado y para quién | B.2: impacto + beneficiario |
| Cliente (10) | Cliente/usuario interno | C: interno |
| | Cliente/usuario externo | C: externo + CRL |
| | Apoyo recibido | E: apoyo recibido |
| Alineamiento (10) | Foco estratégico | D: foco |
| | Criterios "la caja" | derivado por Evaluador |
| Equipo (15) | Equipo interno | E: equipo interno |
| | Equipo externo | E: equipo externo |
| | Sponsor | E: patrocinador |
| | Otros recursos | E: otros recursos |
| Riesgo (20) | Incertidumbre cliente | F + CRL |
| | Incertidumbre solución | F + TRL |
| | Incertidumbre modelo | F + BRL |
| Hitos (10) | Hitos técnicos/operativos | G: técnicos |
| | Hitos económicos | G: económicos |
| | Horizonte de retorno | G: meses → >24=1 / 18-24=3 / <18=5 |

Bandas de conversión nivel→score 1/3/5 (regla de PUNTAJE del Evaluador; el DBI siempre registra el nivel único, nunca la banda):
- TRL → estado de desarrollo: 1-2 = 1 (idea) / 3-6 = 3 (concepto-prototipo) / 7-9 = 5 (validado en operación real).
- CRL → interés / cliente externo: 1-3 = 1 (hipótesis-conversaciones) / 4-5 = 3 (interés concreto declarado) / 6-9 = 5 (validado en prueba o adopción).
- Incertidumbre cliente (invertida): CRL 6-9 = 1 / 4-5 = 3 / 1-3 = 5.
- Incertidumbre solución (invertida): TRL 7-9 = 1 / 3-6 = 3 / 1-2 = 5.
- Incertidumbre modelo (invertida): BRL 6-9 = 1 / 3-5 = 3 / 1-2 = 5.

Notas para el Evaluador:
- "Duración estimada" (E) no se puntúa; informa complejidad.
- Novedad (eje de compuerta 1-5) se deriva del grado del diferenciador: similar=1 / mejora relevante=3 / nuevo=5.
- Índice de Incertidumbre (eje de compuerta) = promedio de las tres incertidumbres. En el scorecard, MÁS incertidumbre puntúa MÁS (la incertidumbre es la materia prima del Sandbox). Compuerta: Novedad ≤2 e Incertidumbre ≤2 → fuera de innovación (derivar a área).
- Compuertas de puntaje total: <60 fuera; >60 con apoyo; >80 Sandbox.

Reglas de plantilla:
- Frases cortas. Números con fuente. Ausente: «No especificado».
- Núcleo faltante (A, B, C, E): marcar PENDIENTE.
- F y G se capturan siempre; dato faltante = «No definido», no se omite el bloque.
- Sin patrocinador: «Sin patrocinador identificado» y marcar en pendientes.
- Tamaño de mercado: factores separados, nunca colapsados en una cifra única.
- TRL/CRL/BRL: SIEMPRE un único número entero 1-9, nunca un rango o banda («5-6» no es válido; ante duda, el menor). Línea de evidencia obligatoria; sin evidencia citable, el nivel baja al que la evidencia sostiene; sin información, «No declarado».
